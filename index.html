Phase · Foundation
00
Seeing What We Have
✓ Complete
Platform is live and deployed on SK's server. All available data integrated. Gap diagnosis produced and documented.
The Experience
PMO opens one screen and sees all of SK's water data for the first time. No more switching between files and tools.
The Big Bet
Unifying fragmented data into one map changes how SK sees and talks about water access — before any intelligence is added.
The Value
Full organizational visibility for the first time. Every data gap named and documented so Phase 1 knows exactly what to fix.
Building Blocks
6 items
  • GIS platform built and deployed on SK's server
  • Data ingestion pipeline for all available datasets
  • Unified spatial database in geographic format
  • Layer toggle system per dataset
  • Demand Score algorithm per community
  • Data quality report per dataset
Key Learnings
4 questions
  • Will the PMO use a map tool daily?
  • Which gaps matter most to fix first?
  • Does the Demand Score match how PMO thinks about urgency?
  • How much does the team trust unverified outputs?
Pre-Requisites
None. Fully internal build. No external agreements needed.
Risks
4 risks
  • Data too incomplete to produce any useful output
  • PMO does not engage and stays with old tools
  • Demand Score does not reflect operational reality
  • Server deployment issues on SK infrastructure
Deliverables
5 artifacts
  • Live GIS platform on SK's server
  • Interactive map with toggleable layers
  • Data quality report per dataset
  • Full gap diagnosis report
Phase · Verification
01
Knowing Where Demand Is Today
→ Next
Development ready to begin. Awaiting data sharing agreements with NWC and NPO providers before work starts.
The Experience
PMO verifies every map layer in working sessions. When done — they trust what they see. NWC updates a station and it appears on the map immediately. No email. No request. No waiting.
The Big Bet
Deep investment in data verification before any intelligence is built produces a foundation every future phase can depend on completely.
The Value
Map reflects ground truth. Demand Score and Heatmap trusted and visible for the first time. Partners manage their own data. Historical records ready to power Phase 2 AI.
Building Blocks
8 items
  • Data sharing agreements signed with NWC and NPO providers
  • ETL pipelines for every new dataset
  • PMO verification sessions per map layer
  • Provider data portal for NWC and NPOs
  • Data quality dashboard — completeness and freshness per dataset
  • Historical records structured as ML training data
  • Demand Score recalculated on verified data
  • Heatmap activated on verified data
Key Learnings
4 questions
  • How long do government data agreements actually take?
  • Which datasets shift demand scores most when verified?
  • Will NPOs maintain their data through the portal?
  • How much does the score change on real vs. assumed data?
Pre-Requisites
Leadership Led
  • NWC data agreement signed — stations, capacity, tanker routes
  • NPO data agreements signed — Erwaa, Al Rajhi, others
  • Current population data
  • Village-level admin boundaries
  • Road network data confirmed
  • PMO commits to two working sessions per month
Risks
6 risks
  • NWC agreement delayed
  • Data received is still incomplete or unusable
  • NPOs refuse to share infrastructure data
  • PMO cannot commit to verification sessions
  • Historical records too sparse to train reliable ML
  • Provider portal not adopted — partners stay on email
Deliverables
6 artifacts
  • Verified map with all data layers confirmed by PMO
  • Demand Score live on verified data
  • Heatmap live on verified data
  • Provider upload portal for NWC and NPOs
  • Data quality dashboard — live completeness view
  • Structured historical records database ready for ML
Phase · Intelligence
02
Choosing the Right Intervention
○ Planned
Scope defined. Begins after Phase 1 is fully complete — all data verified, provider agreements active, historical records confirmed ML-ready.
The Experience
Proposal arrives at 9am. By 9:05am the platform has assessed the site, calculated drive times, analyzed satellite imagery, and recommended a better solution. PMO opens the file, and the work is already half done.
The Big Bet
ML trained on SK's own verified history produces better recommendations than individual judgement — and the PMO will trust and act on them.
The Value
Proposals assessed in minutes not weeks. Field visits become the exception. Every recommendation backed by SK's full project history. Duplicate projects caught before approval.
Building Blocks
8 items
  • ML recommendation engine trained on Phase 1 historical records
  • Proposal submission API integration
  • Automated spatial analysis on every submission
  • Computer vision pipeline on satellite imagery
  • pgRouting drive time on verified road network
  • Project requested vs. recommended comparison view
  • Overlap detection against all provider coverage zones
  • Role-based user access — PMO, leadership, admin
Key Learnings
5 questions
  • How accurate are ML recommendations vs. experienced PMO decisions?
  • Which site factors matter most to actual PMO decisions?
  • How does PMO respond when recommendation differs from NPO proposal?
  • What is the real time saving per proposal?
  • Which request types still need a field visit?
Pre-Requisites
4 items
  • Phase 1 fully and completely done
  • Historical records confirmed ML-ready by dev team
  • PMO sessions scheduled to validate early recommendations
  • Satellite imagery provider agreement confirmed
Risks
5 risks
  • Historical records too thin — recommendations too generic to be useful
  • PMO does not trust automated recommendations
  • Satellite analysis inaccurate in certain terrain types
  • API integration with proposal system blocked
  • Overlap detection misses providers not yet in the portal
Deliverables
7 artifacts
  • Live proposal intelligence pipeline — every submission auto-assessed
  • ML recommendation engine with confidence scoring
  • Satellite site assessment tool
  • Road-based drive time calculator
  • Proposed vs. recommended comparison interface
  • Live overlap detection against all provider zones
  • Role-based access system
Phase · Conversational
03
Knowing Where Demand Is In The Future
◔ Indicative
Features defined but not yet scoped for development. Begins after Phase 2 has run for a minimum operational period and produced enough decision history for the AI to learn from.
The Experience
Team member types in Arabic: "Which villages in Najran have critical demand and no project within 60km?" Answer appears in seconds with map highlights and a priority order. Leadership runs a simulation before a board meeting and walks in with the numbers already done.
The Big Bet
Removing the technical barrier between decision-makers and data — through natural language — expands the platform from a PMO tool to an organization-wide intelligence layer.
The Value
Anyone in SK interrogates the platform without technical skills. Decisions faster with more evidence. Leadership briefs government from one screen. SK acts before crises — not after.
Building Blocks
7 items
  • Project simulation engine with map-based before-and-after view
  • Predictive demand model on population projections and demand history
  • LLM integrated with spatial query translation layer
  • Arabic and English language support throughout
  • Auto-generated proposal briefings
  • Auto-generated simulation and impact reports
  • CEO and leadership national dashboard
Key Learnings
4 questions
  • Which questions do PMO and leadership ask most — does AI answer them reliably?
  • How accurate are demand predictions vs. communities that later file requests?
  • Does simulation change what leadership decides to fund?
  • What AI confidence level is needed before PMO acts without extra verification?
Pre-Requisites
3 items
  • Phase 2 operational for a minimum run period
  • Population growth projections loaded
  • Leadership sessions completed for dashboard design
Risks
5 risks
  • AI assistant gives confident but wrong answers — erodes trust
  • Predictive model produces false signals before enough history has built up
  • Leadership does not engage with dashboard design
  • Partner feeds arrive inconsistently or need heavy manual processing
  • AI struggles with Arabic dialects or water sector terminology
Deliverables
6 artifacts
  • Project simulation tool with before-and-after map view
  • Predictive demand engine with forward-looking risk layer
  • AI conversational assistant — Arabic and English in map interface
  • Auto-generated proposal briefings
  • Auto-generated simulation and impact reports
  • CEO and leadership national dashboard
Phase · Real-Time
04
Seeing What's Happening Now
◌ Vision
Direction confirmed. Detailed scoping begins during Phase 3. IOT pilot must be approved and planned before any national commitment is made.
The Experience
2am. A tank in Najran goes offline. By 2:05am the map has flagged it, recalculated demand scores for eleven affected communities, and sent an alert. Nobody filed a report. Nobody was in the field. The platform noticed first.
The Big Bet
Connecting the platform to real physical infrastructure eliminates the most dangerous gap in SK's operations — the time between when infrastructure fails and when anyone knows about it.
The Value
Failures visible in hours not weeks. No community silently loses water access. Live operational picture of every monitored asset. Resources redirected immediately when failure occurs.
Building Blocks
8 items
  • Live API connections to NWC and partner operational systems
  • IOT sensor integration layer for continuous sensor data streams
  • IOT pilot program in one region before national rollout
  • Infrastructure health map layer — green, amber, red per asset
  • Automated failure detection and alert system
  • Real-time demand score recalculation on source failure
  • Community status update engine — automatic served-to-unserved transitions
  • Hardware procurement, installation, and maintenance planning
Key Learnings
5 questions
  • How quickly can sensor failures be distinguished from real infrastructure failures?
  • Which asset types produce the most actionable data when monitored?
  • How often do alerts produce false positives — and what thresholds minimize noise?
  • Which providers share live data willingly — and which do not?
  • What minimum sensor coverage makes the live map meaningfully more accurate than periodic data?
Pre-Requisites
4 items
  • IOT hardware procurement planned and budgeted
  • Physical access to infrastructure agreed with NWC and NPO partners
  • Live API access agreements with NWC operational systems
  • IOT pilot site selected, scoped, and approved
Risks
6 risks
  • NWC does not agree to live API access — data stays periodic
  • IOT hardware difficult to install or power in remote locations
  • Sensor data quality too poor — too much noise, too many gaps
  • Pilot region not representative of national infrastructure variety
  • Platform cannot handle volume and frequency of live data streams
  • IOT maintenance in remote areas financially unsustainable
Deliverables
7 artifacts
  • Live API connections to NWC and partner systems
  • IOT sensor integration layer
  • Infrastructure health map layer — real-time asset status
  • Automated failure detection and alert system
  • Real-time demand score recalculation engine
  • Community automatic status update system
  • IOT pilot program report with national rollout recommendations
Phase · Evidence
05
Measuring Real Impact
◌ Vision
Concept confirmed. Full scoping begins during Phase 4. Legal and governance review completed before any external access opens.
The Experience
Donor logs in on a Tuesday. Sees the tank they funded 18 months ago — 2.4 million litres dispensed, 680 people served daily, demand score dropped from 84 to 11. They screenshot it and send it to their own board. They did not ask SK for any of this. It was already there.
The Big Bet
Turning live operational data into transparent real-time impact evidence permanently changes SK's relationship with donors — from trust to proof — and compounds in value as the ML engine improves with every measured outcome.
The Value
Every riyal traceable to a measured outcome. Donors verify impact live without waiting for a report. Recommendation engine improves continuously. SK's credibility backed by evidence not self-reporting.
Building Blocks
8 items
  • Impact measurement pipeline — sensor data vs. Phase 2 projections per project
  • Demand score before-and-after engine per completed project
  • Donor impact portal — live usage, beneficiaries, uptime, score change
  • Internal leadership impact dashboard — full portfolio view
  • Government and sector transparency view — national aggregate picture
  • ML retraining pipeline — outcomes fed back into recommendation engine
  • Legal review and governance framework for external access
  • Donor and NPO onboarding process
Key Learnings
5 questions
  • How closely do actual outcomes match Phase 2 simulation projections?
  • Does real-time impact data improve donor retention or funding levels?
  • Which outcome metrics matter most to different donor types?
  • How much does feeding outcomes back into the ML engine improve future recommendations — and over what timeframe?
  • Will government partners use the platform as a planning tool once they have read-only access?
Pre-Requisites
5 items
  • Phase 4 IOT operational across enough projects to produce meaningful data
  • Legal review of data visibility rules completed
  • Donor onboarding process agreed with corporate relations team
  • Governance framework defining access levels per external user type
  • Enough completed projects with sensor coverage to demonstrate impact credibly
Risks
6 risks
  • Phase 4 IOT coverage too sparse for statistically meaningful impact data
  • Actual outcomes differ significantly from projections — undermines confidence in simulation tool
  • Legal review delays or restricts what the donor portal can show
  • Donors do not engage with the portal — prefer traditional reporting
  • ML retraining produces unexpected recommendation changes the PMO cannot explain
  • Government raises concerns about data accuracy before platform is fully validated
Deliverables
7 artifacts
  • Impact measurement pipeline per project
  • Demand score before-and-after dashboard per project
  • Donor impact portal — live usage, beneficiaries, uptime per funded project
  • Internal leadership impact dashboard
  • Government and sector transparency view
  • ML model retraining pipeline on completed outcomes
  • Governance and access control layer per external user type
